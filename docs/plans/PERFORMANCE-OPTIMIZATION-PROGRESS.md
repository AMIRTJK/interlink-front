> Методология и обоснования — в [`docs/PERFORMANCE.md`](../PERFORMANCE.md) (ссылки вида «§7» ведут туда).
> Этот файл — **план + трекер**. После каждой выполненной задачи её статус закрывается здесь.

---

## 🚦 Как этим пользоваться (важно, читать в начале каждой сессии)

**Чтобы дать старт работе**, напишите одно из:
- «Начни фазу N» / «Сделай P3» — беру всю фазу или конкретную задачу по ID.
- **«Продолжай»** / «Continue» — открываю этот файл, нахожу **первую незакрытую** задачу (сверху вниз) и продолжаю с неё.

**Что я делаю по каждой задаче:**
1. Ставлю статус `🔄 IN PROGRESS`, обновляю блок «Текущее положение» вверху.
2. Вношу **только оптимизационные** правки (скорость/ресурсы). НЕ трогаю дизайн, вёрстку и расположение UI; НЕ мигрирую библиотеки; НЕ делаю FSD/200-строк рефакторинг; `GEMINI.md` не применяю (см. «Рамки работ» ниже).
3. **Не запускаю** `npm run build`, проверку типов и браузерные проверки без вашей явной просьбы.
4. Закрываю задачу: `✅ DONE (дата)`, заполняю **Лог** (что изменил, какие файлы, эффект/риск).
5. Обновляю «Общий прогресс» и «Текущее положение».

**Возобновление в новой сессии:** этот файл + запись в памяти (`performance-optimization-roadmap`) позволяют мне поднять контекст и продолжить с нужного места. Просто скажите «продолжай».

**Статусы:** `TODO` · `🔄 IN PROGRESS` · `✅ DONE` · `⏸️ BLOCKED` (жду решения/данных) · `⏭️ SKIP` (решили не делать).
Чекбокс `- [ ]` → `- [x]` дублирует статус для быстрого визуального сканирования.

## 🧭 Рамки работ (scope) — обязательно к соблюдению

- ✅ **Можно:** ленивая загрузка / code splitting, чанки Vite, кэш TanStack Query, мемоизация (`memo`/`useMemo`/`useCallback`), виртуализация, debounce/`useDeferredValue`, оптимизация ресурсов (шрифты/картинки/preload/сжатие), удаление мёртвых зависимостей.
- 🚫 **Нельзя (в этом цикле):** менять дизайн, вёрстку, расположение и внешний вид UI; мигрировать/менять UI-библиотеку; добавлять новые antd-зависимости; делать FSD-разбиение и рефактор «200 строк»; переносить код в `lib.ts`/`model.ts` ради структуры.
- **Ant Design** остаётся как есть — команда уходит от него **отдельно и позже**. Здесь его только оптимизируем, не расширяя связанность.
- **`GEMINI.md` не используется** как источник правил в этих задачах.
- **Инвариант:** внешний вид до и после правок совпадает. Если оптимизация где-то неизбежно затрагивает вёрстку/вид — помечаю задачу `⏸️ BLOCKED` и спрашиваю, а не меняю на своё усмотрение.

## Порядок и принцип

Идём **снизу вверх по стеку зависимостей**: сначала измерение → каркас → общие shared-компоненты (их переиспользуют все) → затем модули по одному от лёгких к тяжёлым. Так каждая правка в общем слое усиливает эффект во всех модулях сразу, а модульные фазы не конфликтуют друг с другом.

> **Правило измерения (§1):** где возможно — фиксируем «до/после» (размер чанка, кол-во ререндеров в Profiler). Реальные прогоны build/Lighthouse — только по вашей команде.
