# Дизайн-документ: Рефакторинг загрузки данных профиля и MFA

## Проблема / Контекст
1. Запрос `GET /api/v1/auth/me` должен выполняться только на уровне страницы «Личный кабинет» (`Profile.tsx`), которая является основным источником данных о пользователе. В данный момент он дублируется при открытии модального окна настроек через компонент `MfaSecurity.tsx`.
2. Страница профиля по-прежнему использует запрос `GET /api/v1/admin/users/1` (`ApiRoutes.FETCH_USER_BY_ID`) для загрузки информации о пользователе. Этот запрос необходимо полностью исключить и заменить на `GET /api/v1/auth/me`.

## Описание изменений

### 1. Перенос запроса в родительский компонент (`Profile.tsx`)
* Заменяем URL `ApiRoutes.FETCH_USER_BY_ID` + `userId` на `ApiRoutes.AUTH_ME` в `useGetQuery` в файле `src/features/Profile/Profile.tsx`.
* Настраиваем устойчивое чтение данных (как `rawUserData?.data`, так и `rawUserData` напрямую для обратной совместимости).
* При получении данных сохраняем их в `tokenControl.setUserData(user)`.
* Передаем статус MFA (`mfaEnabled`), состояние загрузки (`isStatusLoading`) и функцию обновления (`onRefresh`) в модальное окно настроек `ProfileSettingsModal`.
* Удаляем неиспользуемые импорты и комментарии.

### 2. Изменение модального окна настроек (`ProfileSettingsModal.tsx`)
* Принимаем пропсы `mfaEnabled`, `isStatusLoading`, `onRefresh` и передаем их дальше в компонент `MfaSecurity`.
* Комментарии отсутствуют.

### 3. Рефакторинг компонента двухфакторной защиты (`MfaSecurity.tsx`)
* Удаляем локальный вызов `useGetQuery` с URL `ApiRoutes.AUTH_ME` и тип `IMeResponse`.
* Принимаем статус `mfaEnabled`, `isStatusLoading`, `onRefresh` в качестве пропсов.
* После успешной активации или деактивации MFA вызываем проп `onRefresh()` для обновления профиля в родительском компоненте.
* Удаляем все комментарии из файла.
