## 1. Философия: на что мы реально оптимизируем

Производительность — это не «быстрый JS». Это **три отдельных бюджета**, которые нужно держать одновременно:

| Бюджет | Что это | Чем управляем |
|---|---|---|
| **Загрузка** | Сколько байт и запросов до первого полезного экрана | Bundle size, code splitting, сжатие, ресурсы |
| **Отзывчивость** | Задержка на клик/ввод (INP) | Работа главного потока, ререндеры, тяжёлые задачи |
| **Стабильность** | Ничего не «прыгает» (CLS) | Размеры-заглушки, шрифты, изображения, скелетоны |

**Правило №1:** сначала измерь, потом оптимизируй. Не оптимизируй то, что не подтверждено профайлером — это трата времени и источник багов.

**Правило №2:** самая быстрая работа — это работа, которую не делают. Лучший чанк — не загруженный чанк. Лучший ререндер — не случившийся ререндер. Лучший запрос — отданный из кэша.

## 2. Метрики и бюджеты производительности

### Core Web Vitals (цели 2026)

| Метрика | Что измеряет | 🟢 Хорошо | 🔴 Плохо |
|---|---|---|---|
| **LCP** (Largest Contentful Paint) | Когда виден главный контент | ≤ 2.5 с | > 4.0 с |
| **INP** (Interaction to Next Paint) | Отзывчивость на ввод (заменил FID с 2024) | ≤ 200 мс | > 500 мс |
| **CLS** (Cumulative Layout Shift) | Визуальная стабильность | ≤ 0.1 | > 0.25 |
| **TTFB** (Time To First Byte) | Ответ сервера | ≤ 0.8 с | > 1.8 с |
| **FCP** (First Contentful Paint) | Первый рендер | ≤ 1.8 с | > 3.0 с |

> ⚠️ **INP — самая недооценённая метрика в дашбордных SPA вроде нашего.** Тяжёлые таблицы, фильтры,
> редактор и большие ре-рендеры бьют именно по INP. Держите обработчики короткими, разбивайте задачи (см. §14).

### Бюджеты бандла (ориентир для этого проекта)

- **Initial JS (то, что грузится до интерактива), gzip:** ≤ **200 КБ**. Это entry + vendor-критичное. Всё остальное — по требованию.
- **Любой отдельный route-чанк, gzip:** ≤ **150 КБ**.
- **Тяжёлые библиотеки (CKEditor, pdfjs, xlsx, recharts) — 0 КБ в initial.** Только lazy.
- **CSS initial, gzip:** ≤ **50 КБ**.

Зафиксируйте бюджет в CI, чтобы регресс ловился автоматически (см. §3).

```bash
npm i -D rollup-plugin-visualizer
```

```ts
// vite.config.ts
import { visualizer } from "rollup-plugin-visualizer";

plugins: [
  // ...
  visualizer({ open: true, gzipSize: true, brotliSize: true, filename: "dist/stats.html" }),
],
```

Смотрите на treemap: что реально попало в бандл, какие либы дублируются, что тянется в initial-чанк по ошибке.

### 3.2. Реальные метрики пользователей (RUM) через `web-vitals`

```bash
npm i web-vitals
```

```ts
// src/shared/lib/reportWebVitals.ts
import { onLCP, onINP, onCLS, onTTFB, onFCP } from "web-vitals";

export const reportWebVitals = () => {
  const send = (m: { name: string; value: number }) => {
    // отправить на свою аналитику / бэкенд-эндпоинт
    navigator.sendBeacon?.("/metrics/vitals", JSON.stringify(m));
  };
  onLCP(send); onINP(send); onCLS(send); onTTFB(send); onFCP(send);
};
```

Lighthouse — это лабораторные условия. **RUM показывает правду о реальных пользователях** (их сеть, их устройства).

### 3.3. Lighthouse CI в пайплайне

```bash
npm i -D @lhci/cli
```

Настройте пороги (assertions) и запускайте на каждый PR. Регресс производительности должен падать так же, как падают типы или линт.

### 3.4. React DevTools Profiler

- Вкладка **Profiler** → «Record» → выполните сценарий → смотрите, какие компоненты ререндерятся и почему («Why did this render?»).
- Включите **«Highlight updates when components render»** — увидите лишние перерисовки визуально.

### 3.5. Chrome DevTools → Performance

- **Main thread** — ищем «длинные задачи» (Long Tasks > 50 мс) — они убивают INP.
- **Coverage** (Ctrl+Shift+P → «Coverage») — сколько JS/CSS загружено, но не использовано.

## 4. Сборка и Vite

Текущий `vite.config.ts` уже неплох (SWC, `manualChunks`, gzip+brotli). Ниже — как довести до максимума.

### 4.1. Разделение вендоров (`manualChunks`)

**Плохо** — всё в один `vendor` (любое обновление одной либы инвалидирует весь чанк в кэше браузера):

```ts
manualChunks(id) {
  if (id.includes("node_modules")) return "vendor";
}
```

**Хорошо** — стабильное ядро отдельно от остального, тяжёлое — своими чанками (как уже сделано, плюс выделяем react-ядро):

```ts
build: {
  target: "es2022",              // современные браузеры; меньше полифиллов и транспиляции
  cssCodeSplit: true,            // CSS дробится по чанкам (в Vite включено по умолчанию — не отключать)
  reportCompressedSize: false,   // ускоряет сам билд (размеры смотрим через visualizer)
  sourcemap: "hidden",           // карты есть для Sentry, но не палятся в проде
  rollupOptions: {
    output: {
      manualChunks(id) {
        if (!id.includes("node_modules")) return;
        // Ядро, которое меняется реже всего → лучший long-term cache
        if (id.includes("react-dom") || id.includes("/react/") || id.includes("scheduler"))
          return "react-vendor";
        if (id.includes("antd") || id.includes("@ant-design") || id.includes("rc-"))
          return "antd-vendor";
        if (id.includes("ckeditor5")) return "ckeditor-vendor";
        if (id.includes("react-pdf") || id.includes("pdfjs-dist")) return "pdf-vendor";
        if (id.includes("xlsx")) return "xlsx-vendor";
        if (id.includes("mammoth")) return "mammoth-vendor";
        if (id.includes("recharts") || id.includes("d3-")) return "charts-vendor";
        if (id.includes("framer-motion")) return "motion-vendor";
        if (id.includes("@tanstack")) return "query-vendor";
        return "vendor";
      },
    },
  },
},
```

**Почему:** браузер кэширует чанки по имени+хэшу. Если `react`/`antd` лежат отдельно, то релиз с изменением бизнес-логики не заставляет пользователя перекачивать 300 КБ вендоров — они берутся из кэша.

### 4.2. `chunkSizeWarningLimit` — не глушить, а слышать

**Плохо:** `chunkSizeWarningLimit: 2000` — это выключение сигнализации. Предупреждение о чанке в 2 МБ — это не «шум», это баг.

**Хорошо:** держите порог низким (например `600`) и реагируйте на предупреждения делением чанка/ленивой загрузкой.

### 4.3. Minify

`esbuild` (текущий выбор) — правильный дефолт: он в 20–40 раз быстрее terser при почти том же результате. `terser` в devDependencies при этом **не используется** — его можно удалить (см. §19). Оставляйте `esbuild`, если нет доказанной выгоды terser по размеру.

### 4.4. Сжатие

`vite-plugin-compression` (gzip + brotli) уже стоит — отлично. Убедитесь, что **сервер/CDN реально отдаёт `.br`/`.gz`** с правильным `Content-Encoding`. Иначе предсжатые файлы просто лежат мёртвым грузом. Brotli на статике даёт ~15–20% против gzip.

### 4.5. Tree-shaking

- Импортируйте **точечно**, не барелями целиком: `import { Button } from "antd"` — ок (antd v6 — ESM, шейкается). А вот `import * as Icons from "@ant-design/icons"` затянет **все** иконки. Импортируйте каждую иконку по имени.
- В своих `index.ts` (public API модулей) реэкспортируйте только нужное. Барели, реэкспортирующие «всё из всего», ломают tree-shaking и раздувают графы зависимостей.
- Не используйте побочные эффекты на верхнем уровне модулей, которые нельзя выкинуть. Если модуль их не имеет — это работает автоматически при ESM.

### 4.6. `modulePreload`

Vite по умолчанию вставляет `<link rel="modulepreload">` для зависимостей entry. Не отключайте `build.modulePreload`. Это подгружает граф зависимостей параллельно, а не водопадом.

## 5. Code splitting и lazy loading

### 5.1. Роуты — уже сделано правильно

`AppRouter.tsx` грузит все страницы через `React.lazy` + один `<Suspense>`. Это база. Дополнительно:

**Не оборачивайте каждый роут в собственный `<Suspense>` без причины** — в `AppRouter` есть места с вложенными `<Suspense fallback={<Loader fullScreen/>}>` вокруг отдельных роутов. Полноэкранный лоадер на под-роут внутри уже смонтированного layout вызывает мелькание и потерю контекста. Используйте общий Suspense на уровне контента или локальный скелетон вместо `fullScreen`.

### 5.2. Тяжёлые библиотеки — только по требованию

CKEditor, `pdfjs-dist`/`react-pdf`, `xlsx`, `mammoth`, `recharts`, `framer-motion` — это **мегабайты**. Их нельзя импортировать статически в компонентах, которые попадают в initial-граф.

**Плохо:**

```tsx
import * as XLSX from "xlsx"; // весь xlsx (~1 МБ) в чанк страницы, даже если экспорт кликнут раз в месяц

const ExportButton = ({ rows }: IProps) => {
  const onExport = () => {
    const ws = XLSX.utils.json_to_sheet(rows);
    // ...
  };
  return <Button onClick={onExport}>Экспорт</Button>;
};
```

**Хорошо** — динамический импорт в момент действия:

```tsx
const ExportButton = ({ rows }: IProps) => {
  const onExport = async () => {
    const XLSX = await import("xlsx"); // грузится только при клике
    const ws = XLSX.utils.json_to_sheet(rows);
    // ...
  };
  return <Button onClick={onExport}>Экспорт</Button>;
};
```

Аналогично: PDF-просмотрщик, редактор CKEditor, парсер `mammoth` — грузятся только когда пользователь реально открывает соответствующий экран/модалку.

### 5.3. Ленивые модалки, вкладки, дровера

Скрытый по умолчанию UI не должен быть в initial-графе.

```tsx
const HeavyEditorModal = lazy(() =>
  import("./HeavyEditorModal").then((m) => ({ default: m.HeavyEditorModal })),
);

// в рендере:
<If condition={isOpen}>
  <Suspense fallback={<Loader />}>
    <HeavyEditorModal onClose={close} />
  </Suspense>
</If>
```

### 5.4. Prefetch — маскируем задержку lazy-загрузки

`React.lazy` начинает грузить чанк только когда компонент нужен → пользователь видит спиннер. Устраните это **предзагрузкой на намерение** (hover/focus по ссылке, либо `requestIdleCallback` после idle):

```tsx
// фабрика, чтобы переиспользовать один и тот же промис для lazy и для prefetch
const importEditor = () => import("@pages/modules/correspondence/CreateCorrespondencePage");
const CreateCorrespondencePage = lazy(() =>
  importEditor().then((m) => ({ default: m.CreateCorrespondencePage })),
);

const CreateLink = () => (
  <Link to="create" onMouseEnter={() => void importEditor()} onFocus={() => void importEditor()}>
    Создать
  </Link>
);
```

**Почему:** к моменту клика чанк уже в кэше → переход мгновенный, без спиннера. Импорты дедуплицируются браузером, повторная загрузка не происходит.

### 5.5. `React.lazy` только для default/named — избегайте водопадов

Не делайте `lazy` внутри `lazy` без нужды — каждый уровень добавляет round-trip. Плоская схема разбиения по роутам предпочтительнее глубоко вложенной.

## 6. Рендеринг React (React 19 + React Compiler)

Это самый важный раздел для отзывчивости (INP) сложных экранов.

### 6.1. 🔑 React Compiler — главный сдвиг 2026 года

React 19 + **React Compiler** (стабилен с 2025) автоматически мемоизирует компоненты и значения на этапе сборки. Он делает то, что раньше делали руками через `React.memo`/`useMemo`/`useCallback` — но полнее и без ошибок «забыл зависимость».

```bash
npm i -D babel-plugin-react-compiler
```

> ⚠️ Компилятору для работы нужен Babel-проход. Текущий проект собирается на **SWC** (`@vitejs/plugin-react-swc`).
> Варианты: подключить `vite-plugin-react` (Babel) с плагином компилятора, либо дождаться/использовать SWC-интеграцию компилятора.
> Это архитектурное решение — согласуйте с командой. **Пока компилятор не включён — продолжаем следовать ручным правилам мемоизации из `GEMINI.md`.**

**Ключевой принцип 2026:** мемоизация — это оптимизация, а не украшение. Пока нет компилятора — мемоизируйте **точечно и обоснованно** (по данным профайлера), а не «на всякий случай везде». Лишние `useMemo`/`useCallback` тоже стоят памяти и времени сравнения.

### 6.2. Пока компилятора нет: ручная мемоизация правильно

**`React.memo`** — для тяжёлых дочерних компонентов, которые часто получают одни и те же пропсы:

```tsx
// Плохо: тяжёлая строка таблицы перерисовывается при любом ререндере родителя
const TableRow = ({ row, onSelect }: IProps) => { /* сложный JSX */ };

// Хорошо: memo + стабильные пропсы
const TableRow = memo(({ row, onSelect }: IProps) => { /* сложный JSX */ });
```

**`useCallback`** для колбэков, которые уходят вниз в мемоизированные компоненты (иначе memo бесполезен — новый колбэк каждый рендер):

```tsx
// Плохо: новый onSelect каждый рендер → memo(TableRow) всё равно перерисовывается
const onSelect = (id: string) => setSelected(id);

// Хорошо
const onSelect = useCallback((id: string) => setSelected(id), []);
```

**`useMemo`** для дорогих вычислений (сортировки/фильтрации больших массивов, парсинга), **не** для примитивов:

```tsx
// Плохо: useMemo вокруг дешёвой операции — оверхед больше пользы
const title = useMemo(() => `${first} ${last}`, [first, last]);

// Хорошо: мемоизируем реально дорогое
const sortedRows = useMemo(
  () => [...rows].sort((a, b) => a.date.localeCompare(b.date)),
  [rows],
);
```

### 6.3. Стабильные `key` в списках

**Плохо:** `key={index}` — при вставке/удалении/сортировке React путает элементы, теряет состояние, лишние перерисовки/размонтирования.

```tsx
{rows.map((row, i) => <Row key={i} data={row} />)}
```

**Хорошо:** стабильный уникальный id из данных.

```tsx
{rows.map((row) => <Row key={row.id} data={row} />)}
```

### 6.4. Не создавайте объекты/массивы/функции в JSX-пропсах без нужды

**Плохо** — новый объект `style`/`params` каждый рендер ломает memo дочерних:

```tsx
<Child config={{ mode: "edit" }} items={data.filter((x) => x.active)} />
```

**Хорошо** — вынести за пределы рендера или мемоизировать:

```tsx
const activeItems = useMemo(() => data.filter((x) => x.active), [data]);
// config — константа модуля, раз не зависит от пропсов:
const EDIT_CONFIG = { mode: "edit" } as const;
<Child config={EDIT_CONFIG} items={activeItems} />
```

### 6.5. Разделяйте Context, держите state локально

**Плохо:** один «god-context» со всем подряд — любое изменение перерисовывает всех потребителей.

**Хорошо:** дробите контексты по частоте изменения (например, отдельно «тема», отдельно «текущий пользователь», отдельно «фильтры»), а часто меняющееся состояние держите как можно ближе к месту использования (state colocation). Серверные данные вообще не кладите в контекст — их место в TanStack Query (см. §7).

### 6.6. `useTransition` и `useDeferredValue` для тяжёлого UI

Ввод в поиск/фильтр, который перестраивает большую таблицу, лагает, потому что рендер списка блокирует отрисовку ввода. React 19 умеет пометить обновление как несрочное:

```tsx
const [query, setQuery] = useState("");
const deferredQuery = useDeferredValue(query); // список считает по «отложенному» значению

const filtered = useMemo(
  () => rows.filter((r) => r.title.includes(deferredQuery)),
  [rows, deferredQuery],
);
// input использует query → печатается мгновенно; список догоняет без блокировки ввода
```

Для навигаций/переключений вкладок с тяжёлым контентом:

```tsx
const [isPending, startTransition] = useTransition();
const onTab = (key: string) => startTransition(() => setActiveTab(key));
// UI остаётся отзывчивым, старый контент виден, пока грузится новый
```

### 6.7. Неконтролируемые формы и антд Form

Контролируемый инпут вызывает ререндер на **каждое нажатие**. Для больших форм используйте `Form` от antd (он держит значения в своём сторе и не перерисовывает всю форму на каждый ввод) или неконтролируемые поля с `ref`. Не поднимайте состояние одного инпута выше, чем нужно.

### 6.8. React 19: приятные бонусы

- **`ref` как обычный проп** — `forwardRef` больше не нужен, меньше обёрток.
- **Document Metadata** — `<title>`, `<meta>`, `<link>` можно рендерить прямо в компонентах (полезно для SEO под-страниц, см. §16).
- **Actions / `useActionState` / `useOptimistic`** — встроенные оптимистичные обновления и состояние отправки формы без ручного бойлерплейта.

## 7. Управление состоянием и TanStack Query

Правила проекта: GET через `useGetQuery`, мутации через `useMutationQuery`, ключи кэша — в `enum`/`const`. Это хороший фундамент. Ниже — как выжать из кэша максимум.

### 7.1. 🔴 Критично: `staleTime: 0` в `queryClient` — источник лишних запросов

Сейчас `src/shared/lib/queryClient.ts`:

```ts
queries: { retry: 0, staleTime: 0, refetchOnReconnect: true }
```

`staleTime: 0` означает, что **любая** запрошенная ранее данность считается устаревшей мгновенно → повторный `fetch` при каждом маунте компонента, возврате фокуса на вкладку, ремаунте после навигации. Для дашборда это десятки лишних запросов и мельканий.

**Хорошо** — дать данным разумное «время свежести», разделив «свежесть» (`staleTime`) и «время жизни в памяти» (`gcTime`):

```ts
export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: 1,
      staleTime: 60_000,          // 1 мин: не дёргаем сервер зря
      gcTime: 5 * 60_000,         // держим в памяти 5 мин для мгновенного возврата
      refetchOnWindowFocus: false, // в админках фокус-рефетчи чаще мешают, чем помогают
      refetchOnReconnect: true,
    },
  },
});
```

> Тонкая настройка — по типу данных: справочники/permissions → `staleTime` минуты-часы; живые списки → десятки секунд; редко меняющееся → `Infinity` + ручная инвалидация. `useGetQuery` уже кэширует `FETCH_PERMISSIONS` на 5 минут — это правильный паттерн, распространите подход.

### 7.2. `select` — не тащить лишнее в компонент

`select` трансформирует/сужает данные **до** попадания в компонент, и ререндер происходит только если изменился именно выбранный кусок.

```tsx
// Плохо: компонент получает весь ответ, ререндерится на любое изменение любого поля
const { data } = useGetQuery({ url: ApiRoutes.GET_USERS });
const count = data?.items.length;

// Хорошо: подписка только на число
const { data: count } = useGetQuery({
  url: ApiRoutes.GET_USERS,
  options: { select: (res) => res.items.length },
});
```

### 7.3. `placeholderData` / `keepPreviousData` для пагинации и фильтров

При смене страницы/фильтра не показывайте пустой экран и спиннер — держите прошлые данные, пока грузятся новые (в v4 — `keepPreviousData`, в v5 — `placeholderData: keepPreviousData`):

```tsx
useGetQuery({
  url: ApiRoutes.GET_INTERNAL_CORRESPONDENCES,
  params: { page },
  options: { keepPreviousData: true }, // таблица не «схлопывается» при переключении страниц
});
```

### 7.4. Prefetch следующей страницы/деталей

```tsx
// на hover строки списка — предзагрузить деталь, чтобы открытие было мгновенным
const onRowHover = (id: string) =>
  queryClient.prefetchQuery({
    queryKey: [ApiRoutes.GET_INTERNAL_CORRESPONDENCE, id],
    queryFn: () => _axios(`${ApiRoutes.GET_INTERNAL_CORRESPONDENCE}/${id}`),
  });
```

### 7.5. Оптимистичные обновления мутаций

Не ждите ответа сервера, чтобы обновить UI на предсказуемых действиях (пин/архив/чек):

```tsx
useMutationQuery({
  // ...
  onMutate: async (vars) => {
    await queryClient.cancelQueries({ queryKey: [KEY] });
    const prev = queryClient.getQueryData([KEY]);
    queryClient.setQueryData([KEY], (old) => applyOptimistic(old, vars));
    return { prev };
  },
  onError: (_e, _vars, ctx) => queryClient.setQueryData([KEY], ctx?.prev), // откат
  onSettled: () => queryClient.invalidateQueries({ queryKey: [KEY] }),
});
```

### 7.6. Инвалидация точечная, а не «снести весь кэш»

**Плохо:** `queryClient.invalidateQueries()` без ключа — инвалидирует вообще всё, лавина рефетчей.
**Хорошо:** инвалидируйте конкретные ключи (правило проекта уже это требует — соблюдайте строго).

### 7.7. Серверное состояние ≠ клиентское

- **Серверные данные** (списки, детали, справочники) → TanStack Query. Не дублировать в `useState`/контекст.
- **Клиентское UI-состояние** (открыта ли модалка, активная вкладка, драфт формы) → локальный `useState`/`useReducer`, при необходимости лёгкий стор (Zustand). Не тяните Redux ради этого.

### 7.8. Рассмотрите миграцию на TanStack Query **v5**

Проект на v4. v5 меньше по размеру, быстрее, с лучшим API (`useSuspenseQuery`, единый `placeholderData`, упрощённые сигнатуры). Миграция несложная — заложите в бэклог как средне-срочную задачу.

## 8. Виртуализация списков и таблиц

Это ключ к работе с «большим количеством данных» без фризов.

**Проблема:** таблица на 1000 строк = 1000 × (ячейки × компоненты) DOM-узлов. Браузер не может быстро раскладывать/перерисовывать десятки тысяч узлов → фризы, огромный INP, память.

**Правило:** любой список/таблица, где может быть **> ~50–100 строк**, обязан быть виртуализирован (рендерятся только видимые строки + небольшой буфер).

**Варианты:**

- **Ant Design Table** (v6) — используйте встроенную виртуализацию: проп `virtual` + фиксированная высота/`scroll={{ y }}`. Это самый малозатратный путь для нашего `UniversalTable`/`RegistryTable`.
- **`@tanstack/react-virtual`** или **`react-window`** — для кастомных списков/сеток/карточек, чатов, ленты.

```tsx
// Плохо: рендерим все строки разом
<Table dataSource={thousandsOfRows} columns={columns} pagination={false} />

// Хорошо: виртуализация + ограниченная высота вьюпорта
<Table virtual scroll={{ y: 560 }} dataSource={thousandsOfRows} columns={columns} pagination={false} />
```

Дополнительно к виртуализации:

- Не делайте тяжёлых вычислений в `render` ячейки — считайте заранее (в `lib.ts`, как требует архитектура: `columns` вне компонента).
- Мемоизируйте `columns` (или держите их модульной константой) — не пересоздавайте каждый рендер.
- Пагинация/бесконечная прокрутка на сервере предпочтительнее, чем «отдать 10 000 строк и виртуализировать». Меньше данных по сети = быстрее всё.

## 9. Сеть, API и кэширование

### 9.1. Транспорт и заголовки (уровень сервера/CDN)

- **HTTP/2 или HTTP/3** — мультиплексирование убирает водопад запросов.
- **Сжатие ответов API** (gzip/brotli) для JSON.
- **Кэш-заголовки статики:** хэшированные ассеты (`app.[hash].js`) → `Cache-Control: public, max-age=31536000, immutable`. `index.html` → `no-cache` (чтобы всегда получать свежие ссылки на ассеты).
- **`Content-Encoding`** должен реально отдавать предсжатые `.br`/`.gz` (см. §4.4).

### 9.2. Меньше запросов, меньше данных

- **Дедупликация**: TanStack Query сам склеивает одинаковые одновременные запросы — не обходите его прямыми `axios`-вызовами (правило проекта это и запрещает).
- **Пагинация/курсоры** вместо «выгрузить всё». Запрашивайте ровно ту страницу, что видна.
- **Field selection**: если бэкенд умеет — просите только нужные поля для списка (без тяжёлых вложений), а детали грузите отдельным запросом при открытии.
- **Отмена неактуальных запросов**: при быстрой смене фильтров старый запрос не нужен — `AbortController`/`cancelQueries`.

### 9.3. Debounce / throttle ввода

Поиск, автокомплит, «живые» фильтры не должны бить в API на каждое нажатие. В проекте есть `useDebouncedCallback` — используйте его.

```tsx
// Плохо: запрос на каждый символ
onChange={(e) => refetchWith(e.target.value)}

// Хорошо: 300–400 мс тишины
const debounced = useDebouncedCallback(refetchWith, 350);
onChange={(e) => debounced(e.target.value)}
```

### 9.4. Пуш/стриминг вместо поллинга

Если есть «живые» данные (уведомления, чат, статусы) — WebSocket/SSE вместо частого поллинга. Поллинг каждые 3–5 сек на десятке экранов = постоянная нагрузка и разряд батареи.

## 10. Загрузка ресурсов и critical path

### 10.1. Resource hints

В `index.html` / динамически:

```html
<!-- Заранее установить соединение с API-доменом (DNS+TCP+TLS) -->
<link rel="preconnect" href="https://api.example.com" crossorigin />
<link rel="dns-prefetch" href="https://api.example.com" />

<!-- Предзагрузить критичный ресурс LCP (например, фон конкретно страницы логина) -->
<link rel="preload" as="image" href="/bg-auth.jpg" fetchpriority="high" />
```

- `preconnect` — для доменов, к которым точно пойдём (API, CDN шрифтов если внешний).
- `preload` — **только** для ресурсов критического пути. Переусердствуете — заберёте пропускную способность у действительно нужного.
- `fetchpriority="high"` — на LCP-ресурс; `="low"` — на явно второстепенное.

### 10.2. `preload` фона авторизации — сейчас грузится всем и всегда

В `index.html` стоит `<link rel="preload" as="image" href="bg-auth.jpg" />`. Этот фон нужен **только** на странице логина, а предзагружается для **каждого** пользователя, включая уже залогиненных, которые на логин не попадут. Это трата трафика на критическом пути.

**Хорошо:** убрать глобальный preload и предзагружать фон **внутри** ленивого чанка `Auth`, либо оставить, но с `fetchpriority` и только если стартовый роут — логин.

### 10.3. `defer`/`async` и порядок

Vite сам вставляет `<script type="module">` (по семантике = `defer`), так что entry не блокирует парсинг HTML. Не добавляйте синхронных `<script>` в `<head>`.

### 10.4. Приоритет критического CSS

Vite инлайнит критический CSS и грузит остальной по чанкам. Не тащите весь глобальный CSS в один огромный файл — держите модульные `style.css` рядом с компонентами (правило проекта), чтобы CSS дробился вместе с JS-чанками.

## 11. Изображения

Изображения — частая причина плохого LCP и CLS.

### 11.1. Современные форматы

**Плохо:** PNG/JPG везде.
**Хорошо:** **AVIF** (лучшее сжатие) с фолбэком на **WebP**, JPG/PNG — крайний фолбэк.

```html
<picture>
  <source srcset="/hero.avif" type="image/avif" />
  <source srcset="/hero.webp" type="image/webp" />
  <img src="/hero.jpg" alt="..." width="1280" height="720" loading="lazy" decoding="async" />
</picture>
```

### 11.2. Всегда указывайте размеры → нет CLS

**Плохо:** `<img src="a.jpg" />` — до загрузки высота 0, потом контент «прыгает» (CLS).
**Хорошо:** `width`/`height` (или `aspect-ratio` в CSS) зарезервируют место заранее.

### 11.3. Lazy vs eager

- Изображения ниже сгиба: `loading="lazy"`.
- **LCP-изображение (главное, на первом экране): `loading="eager"` + `fetchpriority="high"`.** Не лениво! Иначе LCP проседает.

### 11.4. Responsive `srcset`/`sizes`

Не грузите картинку 2000px в контейнер 400px. Отдавайте несколько размеров и дайте браузеру выбрать под DPR и вьюпорт.

### 11.5. Иконки — SVG, не шрифт и не PNG

Иконки — через `@ant-design/icons`/`lucide-react` (SVG-компоненты, шейкаются по одной). Не подключайте icon-font (грузит весь набор) и не плодите PNG.

## 12. Шрифты

### 12.1. 🔴 Сейчас грузится 5 статических начертаний Inter

`src/app/main.tsx` импортирует:

```ts
import "@fontsource/inter/300.css";
import "@fontsource/inter/400.css";
import "@fontsource/inter/500.css";
import "@fontsource/inter/600.css";
import "@fontsource/inter/700.css";
import "@fontsource/inter/800.css";
```

Это **6 начертаний**, каждое — с латиницей **и** кириллицей. Все грузятся при старте, до первого экрана. Итог — лишние сотни КБ шрифтов на критическом пути и риск FOIT/FOUT.

**Хорошо** — вариативный шрифт (один файл на все веса) + подмножество + `font-display: swap`:

```bash
npm i @fontsource-variable/inter
```

```ts
// подключаем ОДИН вариативный файл вместо шести статических
import "@fontsource-variable/inter";
```

Дальше:

- **`font-display: swap`** — текст виден сразу системным шрифтом, потом подменяется (нет «невидимого текста»). У `@fontsource` это обычно дефолт — проверьте.
- **Subset кириллицы + латиницы**, без греческого/вьетнамского и т.п. `@fontsource` даёт файлы по подмножествам; подключайте только `cyrillic` + `latin`.
- **`unicode-range`** — чтобы браузер грузил кириллический файл только когда на странице есть кириллица.
- **Preload только критичного начертания** (обычно 400 и 500), остальное — по мере надобности.

**Почему:** шрифты — блокирующий рендер текста ресурс. Один вариативный файл нужного подмножества с `swap` вместо шести статических = меньше байт, быстрее FCP/LCP, нет мельканий.

## 13. CSS и Tailwind

### 13.1. Tailwind 4 — генерируется только используемое

Tailwind 4 (`@tailwindcss/vite`) генерирует классы по факту использования. Держите правильный охват контента (`content`) в конфиге — не сужайте так, что нужные классы выпадают, и не раздувайте так, что сканируется мусор. **Не пишите огромных `@layer`-простыней «на будущее»** — это мёртвый CSS.

### 13.2. `content-visibility` для длинных страниц

Секции ниже вьюпорта можно не раскладывать, пока не приблизились:

```css
.section-offscreen {
  content-visibility: auto;
  contain-intrinsic-size: 0 600px; /* резерв места, чтобы не было CLS */
}
```

**Почему:** браузер пропускает layout/paint невидимых секций → быстрее первый рендер длинных списков/страниц.

### 13.3. `contain`

Для независимых виджетов (карточки, панели), чьи изменения не должны влиять на раскладку соседей:

```css
.widget { contain: content; } /* изоляция layout/paint — меньше площадь пересчёта */
```

### 13.4. Избегайте дорогих селекторов и глубокой вложенности

Тяжёлые универсальные селекторы (`* { }`), глубокие каскады и `!important`-войны увеличивают время стилизации и провоцируют reflow. Tailwind это в основном снимает — держитесь утилит.

### 13.5. Не анимируйте свойства, вызывающие layout

`width`/`height`/`top`/`left`/`margin` в анимации = reflow каждый кадр. Анимируйте `transform` и `opacity` (композитятся на GPU) — см. §15.

## 14. JavaScript и главный поток

INP страдает, когда обработчик или рендер держит главный поток дольше ~50 мс.

### 14.1. Разбивайте длинные задачи

**Плохо:** синхронно обработать 10 000 элементов в обработчике клика — UI замерзает.

**Хорошо:** уступайте потоку (`await scheduler.yield?.()` / `setTimeout(0)` / чанками), либо перенесите в Web Worker.

### 14.2. Web Workers для тяжёлого: xlsx, pdf, парсинг

Парсинг/генерация Excel (`xlsx`), рендер/парс PDF (`pdfjs`), конвертация `mammoth` — это CPU-тяжёлые операции, которые **замораживают UI**, если делать их в главном потоке.

```ts
// Плохо: XLSX.read больших файлов в главном потоке → фриз на секунды
const wb = XLSX.read(data);

// Хорошо: в воркере (например, через Comlink), UI остаётся отзывчивым
const worker = new Worker(new URL("./xlsx.worker.ts", import.meta.url), { type: "module" });
```

> `pdfjs-dist` уже использует worker — убедитесь, что `workerSrc` настроен и воркер грузится (а не фолбэчит в fake worker в главном потоке).

### 14.3. Пассивные слушатели скролла/тача

```ts
// Хорошо: не блокируем скролл
el.addEventListener("scroll", onScroll, { passive: true });
```

### 14.4. `requestIdleCallback` для некритичной работы

Аналитику, префетчи, прогрев кэша делайте в простое, а не конкурируя с рендером первого экрана.

### 14.5. Избегайте layout thrashing

Не чередуйте чтение (`offsetHeight`, `getBoundingClientRect`) и запись (изменение стилей) в цикле — каждый цикл форсит синхронный reflow. Сначала все чтения, потом все записи (batch).

### 14.6. Следите за размером зависимостей

Перед добавлением библиотеки проверьте её вес (bundlephobia/visualizer) и наличие ESM/tree-shaking. Часто нужное можно сделать 20 строками вместо 200 КБ-зависимости. `moment` → `dayjs` (у вас уже `dayjs` — отлично, не возвращайте moment). Не тяните lodash целиком — точечные `lodash-es/<fn>` или нативные методы.

## 15. Анимации

`framer-motion` мощный, но легко ударить по производительности.

- **Анимируйте только `transform` и `opacity`** — они идут на GPU-композит, не вызывают reflow/repaint слоёв.
- **Не анимируйте `width/height/top/left`** — layout каждый кадр = дёрганье.
- **`will-change` точечно** и временно — не вешайте на всё подряд (жрёт память слоёв).
- **Уважайте `prefers-reduced-motion`** — отключайте/упрощайте анимации (это ещё и пункт Accessibility в Lighthouse):

```tsx
const reduce = useReducedMotion(); // из framer-motion
<motion.div animate={reduce ? {} : { opacity: 1, y: 0 }} />
```

- Тяжёлый `framer-motion` (§4.1 — свой чанк `motion-vendor`) не должен попадать в initial, если анимации только на внутренних экранах. По правилам проекта запрещён hover-сдвиг `translateY` у карточек — это и для перфа хорошо (меньше композитных перерисовок).

## 16. Lighthouse: как получить максимальные баллы

### Performance (цель 90–100)

Сумма всего вышеописанного. Ключевое для баллов:

- **LCP ≤ 2.5 с**: критичный ресурс (шрифт/изображение/hero) — preload + `fetchpriority=high`; никакого lazy на LCP-элемент.
- **TBT/INP**: минимум JS в initial (§4, §5), разбитые задачи (§14), виртуализация (§8).
- **CLS ≈ 0**: размеры у изображений (§11.2), `font-display: swap` + резерв (§12), скелетоны фиксированного размера вместо «прыгающего» контента.
- **«Reduce unused JavaScript/CSS»**: code splitting + tree-shaking + удаление мёртвых зависимостей (§19).
- **«Serve images in next-gen formats»**: AVIF/WebP (§11.1).
- **«Enable text compression»**: brotli/gzip реально отдаётся (§4.4, §9.1).
- **«Efficient cache policy»**: `immutable` на хэшированные ассеты (§9.1).

### Accessibility (цель 100)

- Все `<img>` — с осмысленным `alt` (декоративные — `alt=""`).
- Достаточный **контраст** текста (WCAG AA ≥ 4.5:1).
- Все интерактивные элементы **доступны с клавиатуры**, видимый `:focus`.
- Иконки-кнопки без текста — с `aria-label` (у нас они в `<Tooltip>` — добавьте и `aria-label`).
- Корректная иерархия заголовков, `<html lang="ru">` (уже есть), landmark-роли.
- `label`↔`input` связаны; ошибки форм объявляются ассистивным технологиям.
- `prefers-reduced-motion` уважается (§15).

### Best Practices (цель 100)

- Только **HTTPS**, без mixed content.
- Нет ошибок в консоли на проде.
- Актуальные зависимости без известных уязвимостей (`npm audit`).
- Корректный `Content-Security-Policy`, `charset`, `viewport` (есть).
- Изображения в правильном соотношении сторон (не растянуты).

### SEO (цель 100 — насколько применимо для внутренней ЭДО-системы)

- Уникальные `<title>` и `<meta name="description">` на ключевых публичных страницах. В React 19 метаданные можно задавать прямо в компоненте страницы (§6.8).

## 17. Обязательные правила для новых компонентов

Соблюдать при разработке **каждого** нового компонента (в дополнение к архитектурным правилам `GEMINI.md`):

1. **Ленивая загрузка тяжёлого.** Любой компонент, тянущий CKEditor / pdf / xlsx / mammoth / recharts / крупную модалку, подключается через `React.lazy` + `<Suspense>` или динамический `import()` в момент действия. Никогда — статическим импортом в always-mounted коде.
2. **Стабильные `key`.** В списках — id из данных, не индекс.
3. **Нет создания объектов/функций/массивов в JSX-пропсах** без мемоизации, если они уходят в мемоизированные/тяжёлые дети (`useMemo`/`useCallback`/константа модуля). Пока не подключён React Compiler.
4. **`React.memo` для тяжёлых детей**, получающих стабильные пропсы; колбэки к ним — через `useCallback` (это уже требование `GEMINI.md`).
5. **`columns` таблиц — вне компонента** (в `lib.ts`), мемоизированы. Никаких пересозданий каждый рендер (требование архитектуры).
6. **Списки/таблицы > ~50 строк — виртуализированы** (antd `virtual` или `@tanstack/react-virtual`).
7. **Данные — из TanStack Query**, с осознанным `staleTime`; не дублировать серверные данные в локальный стейт; мутации инвалидируют точечные ключи.
8. **`select`** в запросах, если компоненту нужен кусок ответа, а не весь.
9. **Ввод, бьющий в API/тяжёлый рендер, — через debounce** (`useDebouncedCallback`) и/или `useDeferredValue`.
10. **Изображения:** заданы `width`/`height` (или `aspect-ratio`), `loading="lazy"` (кроме LCP), современный формат, осмысленный `alt`.
11. **Анимации:** только `transform`/`opacity`; уважать `prefers-reduced-motion`.
12. **Никаких тяжёлых синхронных вычислений в обработчиках/рендере** — мемоизация, чанкинг или Web Worker.
13. **Доступность с клавиатуры и `aria-label`** на иконочных кнопках; видимый фокус.
14. **Условный рендеринг — через `<If>`** (требование проекта), тяжёлое скрытое содержимое не монтируется, пока не нужно.
15. **Проверить вес** любой новой зависимости перед добавлением (ESM + tree-shaking обязательны).
