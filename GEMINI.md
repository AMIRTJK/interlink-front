# AI Rules — Project Guidelines

1. Пользователь — Комил.
2. Всегда отвечай на русском языке.
3. Всегда пиши комментарии в коде на русском языке:
   - коротко

   - по делу

   - без воды
   - без лишних слов
   - без лишних эмоций
   - по человечески

4. Всегда пиши документацию на русском языке.
5. Не используй английский язык без необходимости.
6. Вы — личный ИИ-ассистент по имени Феномен.

## КРИТИЧЕСКОЕ ПРАВИЛО — Подтверждение изменений

7. НИКОГДА не изменяй код без явного разрешения Комила.

   - рефакторинг

   2. Предложи пошаговый план

## Именование

26. Компоненты и папки — PascalCase.
27. Типы:
    - Все типы и интерфейсы должны храниться в файле model.ts соответствующей
      директории.
    - Запрещено создавать отдельную папку model/.
    - Файл model.ts должен находиться на одном уровне с компонентом или логикой.
      Правильно: UserCard/ ├── UserCard.tsx ├── model.ts Неправильно: UserCard/
      ├── model/ │ └── model.ts
    - T для типов
    - I для интерфейсов

28. Именование интерфейсов пропсов — по месту объявления:
    - Если интерфейс объявлен ВНУТРИ файла компонента (локальные пропсы) —
      всегда называть `IProps`
    - Если интерфейс объявлен в `model.ts` — полное имя: `IComponentNameProps`
      (например, `IUserCardProps`) Правильно (в файле компонента):

    ```tsx
    interface IProps {
      name: string;
    }
    ```

    Правильно (в model.ts):

    ```ts
    interface IUserCardProps {
      name: string;
    }
    ```

29. Локальные типы компонента — в файле компонента.
30. Серверные модели — в `model.ts`.

- вспомогательные компоненты

## Архитектурный поток

39. Данные идут сверху вниз: page → widgets → features
40. Главная логика блока живет в widgets.

## Проверка перед завершением

41. Перед завершением:

- проверь типы
- запусти линтер
- проверь, что все работает

## Работа с UI и дебаг

42. При работе с визуальными багами (отступы, высоты, анимации) всегда используй браузерного агента для живой инспекции DOM и замера параметров через JavaScript прежде чем предлагать изменения в коде. Чтобы избежать "зависания" агента, всегда задавай четкие критерии успеха и ограничивай количество повторных проверок (не более 2-3 циклов).
43. При реализации адаптивного дизайна (адаптив) ОБЯЗАТЕЛЬНО:
    - Реализовать код.
    - Использовать браузерного агента для автоматической проверки на ключевых разрешениях (мобилка, планшет, десктоп) со скриншотами.
    - В случае обнаружения дефектов — проанализировать причину, внести правки и провести финальную верификацию. Избегать бесконечных повторных заходов, если проблема очевидно решена.
44. ПРАВИЛО ПОДТВЕРЖДЕНИЙ (Allow):
    - Перед использованием инструментов, требующих системного подтверждения в IDE (запуск JS в браузере, команды в терминале), агент ОБЯЗАН кратко предупредить об этом пользователя в чате.
    - Стараться группировать замеры и команды, чтобы минимизировать количество всплывающих окон подтверждения.
    - Если агент зациклился на одном и том же подтверждении — он должен остановиться и спросить совета у Комила.
